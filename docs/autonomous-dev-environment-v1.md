# 自律開発環境 v1 標準設計書

## 1. Purpose

本ドキュメントは、複数の開発案件を人手による都度指示なしで、n8nを司令塔とした半自律ワークフローで回すための「自律開発環境 v1」の全体設計を、1枚で把握できる形にまとめるものである。

目的は以下の3点に限定する。

- AI間(Perplexity / Claude Code / Antigravity)の手動橋渡しを減らすための構造を明文化する
- 各ツールの責任範囲と、人間が介入すべき境界(承認ゲート)を明確にする
- 単一案件(Slack請求書プロジェクト)での検証を、他案件へ横展開できる形で設計する

個別のワークフロー実装手順は [workflow-standard.md](workflow-standard.md) に委譲し、本書はその上位にある運用設計・責任分界のみを扱う。

## 2. Scope

対象とするのは「n8nが起点となり、複数のAIツールを介して開発タスクを半自律で進行させる仕組み」全体である。

含むもの:

- n8nをハブとしたタスクの受け渡し・状態管理の設計方針
- 各ツールの役割分担
- 高リスク変更に対する承認フロー
- Slack請求書プロジェクトをPoCとして扱う際の位置づけ

含まないもの(詳細は[10. Out of Scope](#10-out-of-scope)参照):

- 各ツールの個別実装・API仕様
- n8nワークフローのノード単位の設定手順
- 役割ごとの詳細な権限定義(→ [roles-and-responsibilities.md](roles-and-responsibilities.md))
- 個別の承認基準の閾値(→ [approval-rules.md](approval-rules.md))

## 3. Design Principles

1. **司令塔は常に単一**: 複数AIが並行して動く場合でも、状態とタスクの真実はn8nが保持する。AI同士が直接連携し状態を持つ構造にはしない。
2. **人間の介入点は事前定義する**: 「都度判断」ではなく、[6. Approval Gates](#6-approval-gates)で列挙した条件に該当する場合にのみ人間へエスカレーションする。
3. **半自律であって全自律ではない**: v1では高リスク領域を人間承認必須とし、自律範囲を段階的に広げる前提とする(方針は [project-constitution.md](project-constitution.md) に準拠)。
4. **1案件で検証してから横展開する**: 新しい設計要素はまずSlack請求書プロジェクトで検証し、汎用構造として抽象化できたもののみを他案件に適用する。
5. **決定と変更は記録に残す**: 本書に基づく設計判断の変更は [decision-log.md](decision-log.md) に、実運用上の変更は [change-log.md](change-log.md) に記録し、本書自体の逐次改訂では追わない。

## 4. System Overview

```
                        ┌─────────────┐
                        │   Human     │  承認 / 例外対応
                        └──────┬──────┘
                               │ 承認ゲート(6章)
                               ▼
        ┌───────────────────────────────────────┐
        │                  n8n                    │
        │   (司令塔: タスク管理・状態保持・ルーティング) │
        └───┬───────────┬───────────┬─────────────┘
            │            │           │
            ▼            ▼           ▼
      ┌───────────┐ ┌───────────┐ ┌────────────┐
      │Perplexity │ │Claude Code│ │Antigravity │
      │ (調査/情報 │ │(実装/コード│ │(実行環境/  │
      │  収集)    │ │  変更)    │ │ 検証)      │
      └───────────┘ └───────────┘ └────────────┘
            │            │           │
            └────────────┴───────────┘
                         │
                         ▼
                  ┌─────────────┐
                  │   Slack     │  進捗通知 / 人間への
                  │             │  エスカレーション窓口
                  └─────────────┘
```

n8nがすべてのAIツールに対してタスクを発行し、結果を回収する中心ノードとなる。AI間の直接連携は行わず、n8nを経由した非同期の受け渡しに統一する。Slackは通知・エスカレーション専用のインターフェースであり、タスクの実行主体ではない。

## 5. Tool Responsibilities

| ツール | 役割 | 持たないもの |
|---|---|---|
| **n8n** | タスクの受付・状態管理・各ツールへのルーティング・承認ゲートの起動 | コード生成能力、コード実行環境 |
| **Perplexity** | 外部情報の調査・要約・技術選定の根拠収集 | コード変更権限、実行環境へのアクセス |
| **Claude Code** | 設計書・コードの読解と変更、PR相当の変更案の作成 | 本番環境への直接デプロイ権限 |
| **Antigravity** | コード変更の実行・検証(テスト実行、動作確認) | 承認なしでの本番反映 |
| **Slack** | 進捗の可視化、人間への通知、承認依頼の受付窓口 | タスクの実行・判断 |
| **Human** | 承認ゲートでの最終判断、例外対応、方針変更の決定 | 通常フローでの逐次介入(必要時のみ) |

各ツールの詳細な権限・責任の定義(誰が何に対してRACIを持つか)は [roles-and-responsibilities.md](roles-and-responsibilities.md) を正とする。本表はツール選定の全体像を示すに留める。

## 6. Approval Gates

以下に該当する変更は、n8nのフローが自動で完了させず、Slack経由で人間の承認を必須とする。

- 本番環境・顧客向けシステムへの反映を伴う変更
- 認証情報・APIキー・アクセス権限に関わる変更
- 既存の承認ルール・ワークフロー標準自体の変更
- 外部への送信を伴う操作(顧客への通知、請求書送付など)
- ロールバックが困難、または影響範囲が複数案件に及ぶ変更

承認ゲートの具体的な判定基準・閾値・エスカレーション手順は [approval-rules.md](approval-rules.md) に定義されたものを正とし、本書では「ゲートが存在する位置」のみを示す。

## 7. Standard Workflow

1. n8nがタスクを受領し、対象案件・タスク種別を判定する
2. 調査が必要な場合はPerplexityへ、実装が必要な場合はClaude Codeへタスクを発行する
3. Claude Codeが変更案を作成し、Antigravityが実行・検証を行う
4. 検証結果がn8nに戻り、[6. Approval Gates](#6-approval-gates)の条件に該当するか判定する
5. 該当する場合はSlack経由で人間に承認を依頼し、承認後に反映する
6. 該当しない場合はn8nが完了処理を行い、Slackへ進捗を通知する

タスク種別ごとの詳細な分岐・処理手順は [workflow-standard.md](workflow-standard.md) を正とする。本章はゲートがどの段階に挿入されるかを示す骨格のみを扱う。

## 8. Slack Invoice PoC

Slack請求書プロジェクトを、本設計のv1における最初のPoC対象と位置づける。

選定理由:

- タスクの入口(Slack)と出口(請求書送付という高リスク操作)が明確で、[6. Approval Gates](#6-approval-gates)の検証に適している
- 案件規模が小さく、n8n中心構造の検証コストが低い
- 「外部への送信を伴う操作」という承認必須カテゴリを実運用で確認できる

PoCで確認する項目:

- n8nが単一の司令塔として機能し、AI間の直接連携が発生しないこと
- 請求書送付という高リスク操作が、必ずSlack経由の人間承認を経由すること
- PoCで得られた構造(タスク受け渡し・承認ゲートの実装パターン)が、他案件に抽象化して適用可能であること

PoC固有の実装詳細(n8nワークフローの具体ノード構成等)は本書の対象外とし、別途実装ドキュメントで管理する。

## 9. Rollout Plan

- **v1**: Slack請求書プロジェクト1件でPoCを実施し、本書の設計が実運用で成立するかを検証する
- **v1→v2移行判定**: PoCで[6. Approval Gates](#6-approval-gates)が機能し、n8n中心構造が崩れなかったことを確認できた時点で、2件目以降の案件への横展開を開始する
- **v2以降**: 複数案件を並行して扱うための、案件間のリソース競合・優先度制御の設計を追加する

## 10. Out of Scope

以下はv1の対象外とし、将来必要になった時点で別ドキュメントとして起こす。

- 複数案件が同時に走る場合のリソース競合・優先度制御
- 自律範囲の段階的拡大(人間承認を要するゲートを段階的に減らす基準)
- 各ツールのコスト管理・利用量の最適化
- n8n以外のオーケストレーションツールへの置き換え検討
- 障害時の自動リカバリ・ロールバック手順の詳細設計

## v1でやること / v2以降に回すこと

**v1でやること**

- n8nを唯一の司令塔とする構造を確立する
- ツールごとの責任範囲(5章)と承認ゲート(6章)を明文化する
- Slack請求書プロジェクト1件でPoCを実施し、構造の妥当性を検証する
- PoCで得た知見を、横展開可能な形に抽象化する

**v2以降に回すこと**

- 複数案件の並行運用とリソース・優先度の調整
- 承認ゲートの段階的縮小(自律範囲の拡大)基準の設計
- 障害時の自動リカバリ・ロールバック設計
- ツール構成そのものの見直し(n8n以外への置き換え検討を含む)
