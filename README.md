# Luvira AI DevFlow

Luvira AI DevFlow は、Human / AI / 自動化基盤の役割を明確に分離しながら、実装、レビュー、承認、外部SaaS導入までを一貫して扱うための標準ワークフローである。

## 最初に読むべきドキュメント

1. `docs/project-constitution.md`
2. `docs/workflow-standard.md`
3. `docs/roles-and-responsibilities.md`
4. `docs/approval-rules.md`
5. `docs/decision-log.md`

## ドキュメント構成

- `docs/project-constitution.md` — プロジェクトの憲章、禁止事項、要承認事項の考え方。
- `docs/workflow-standard.md` — 標準ワークフローの正本。
- `docs/roles-and-responsibilities.md` — 役割分担の正本。
- `docs/approval-rules.md` — 承認が必要な変更カテゴリの正本。
- `docs/decision-log.md` — 運用上の判断と設計判断の記録。
- `docs/change-log.md` — 文書変更と運用更新の記録。
- `docs/archive/` — 標準基盤の正本ではない旧設計書や案件依存ドキュメントの退避先。

`README.md` の役割表は `docs/roles-and-responsibilities.md` の全体像表と同一内容を維持する。役割分担の変更は `docs/roles-and-responsibilities.md` を正本として更新し、README 側にも同期する。

## 役割分担

| 役割 | 主担当 | 担当しないこと |
|---|---|---|
| Human | 目的、制約、禁止事項、成功条件を定義する。実装指示を承認する。高リスク変更の GO / NO-GO を承認する。節目レビューのスキップ可否を承認する。人間介入が必要なログイン、2FA、CAPTCHA、決済、法的同意を行う。 | 実装の丸投げ。差戻し論点の一次整理。日常的な実装作業。 |
| Perplexity | 差戻し観点の一次整理を行う。承認に必要な論点を整理する。高リスク変更の論点を Human が承認しやすい形に整理する。外部SaaS導入時は停止条件と完了条件の整理を支援する。 | 実装指示の最終承認。設計レビューの最終責任。実装そのもの。 |
| GSPA | 機能ブロック単位または日次の節目で上流レビューを行う。設計レベルの懸念や構造上の違和感を指摘する。外部SaaS導入フローの設計レベルの懸念をレビューする。 | 差戻し観点の一次整理。通常の実装判断の肩代わり。Human を経ない最終承認。 |
| GAIC | 実装・修正・検証・反映を行う。承認済み方針に従って変更を進める。外部SaaS導入フローに必要な設定反映や後続作業を行う。 | 差戻し観点の整理。承認の代行。設計レビューの最終判断。 |
| Claude | 既存repo理解、流用候補整理、実装支援を行う。必要に応じて外部SaaS導入の事前調査や操作手順の下書きを支援する。 | 実装指示の承認。高リスク変更の GO / NO-GO 判断。差戻し観点の一次整理の正本担当。 |
| n8n | 橋渡し・通知・承認フローを担当する。外部SaaS導入ではブラウザ自動化エージェントの起動、停止、再開、通知を担当する。Human 介入点を明示し、完了条件に達したら結果を返す。 | 設計判断そのもの。承認の代行。差戻し論点の整理。 |
| Browser Automation Agent | GUI依存タスクの画面操作を担当する。ログイン後の遷移、入力、待機、公開URL取得などを自動実行する。Human 介入完了後に同一セッションで処理を再開する。 | クレジットカード入力、法的同意、2FA など Human 専用操作の代行。承認判断。判断基準の変更。 |
| Antigravity | 実装ワーカーとして必要な作業を行う。承認済み方針に従って実装を補助する。 | 承認。設計判断の正本化。差戻し観点の整理。 |

## ワークフローの考え方

- Human は承認と責任の主体であり、実装の丸投げ先ではない。
- Perplexity は差戻し観点の一次整理と承認論点の整理を担う。
- GSPA は節目で原則自動起動する上流レビュー担当であり、Perplexity を経ずに並列起用しない。
- GUI依存の外部SaaS導入は例外作業ではなく、標準ワークフローの正式な対象として扱う。

## 外部SaaS導入の標準ルール

外部サービス導入では、n8n がブラウザ自動化エージェントの起動、停止、再開、通知を制御する。Human 介入が必要なのは、ログイン、2FA、CAPTCHA、決済、法的同意などに限定する。

Railway の導入タスクでは、対象 GitHub リポジトリのデプロイ完了後に Public Domain を取得し、公開URLを返した時点で完了とみなす。200 応答確認や画面表示確認は別タスクとして扱う。

## 旧ドキュメントの扱い

案件専用の強い依存を持つ旧設計書は、標準基盤の正本に含めない。必要に応じて `docs/archive/` に移動し、README の正本一覧には含めない。
